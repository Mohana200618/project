## File Naming Convention

SM-FET uses the following naming convention for all captured evidence:

### Format
```
{PLATFORM}_{EVIDENCE_TYPE}_{DATE}_{TIME}.png
```

### Examples
- `facebook_profile_2025-02-11_10-30-45.png`
- `twitter_posts_2025-02-11_14-22-18.png`
- `instagram_followers_2025-02-11_09-15-33.png`
- `linkedin_messages_2025-02-11_16-45-20.png`

### Components
- **PLATFORM**: facebook, twitter, instagram, tiktok, linkedin
- **EVIDENCE_TYPE**: profile, posts, messages, followers
- **DATE**: YYYY-MM-DD format
- **TIME**: HH-MM-SS format (24-hour)

## Best Practices for Case Organization

### 1. File Naming
- Always use the standard format: `{PLATFORM}_{TYPE}_{DATE}_{TIME}.png`
- Never rename evidence files after capture
- Keep timestamps accurate (UTC preferred)

### 2. Folder Structure
- Create folders immediately when case is created
- Never move evidence files between folders after capture
- Maintain separate folders for each evidence type
- Keep all supporting documents in case root

### 3. Documentation
- Update chain of custody for each capture
- Record all hash values immediately
- Generate hashes before evidence is stored long-term
- Maintain backup copies of all documentation

### 4. Storage
- Store in secure location with limited access
- Use redundant backup systems
- Keep original and backup in separate locations
- Document all storage locations

### 5. Access Control
- Limit access to authorized personnel only
- Log all access to evidence files
- Maintain audit trail of modifications
- Restrict write access after case closure

## Verification Procedures

### Hash Verification Checklist
- [ ] All evidence files hashed with SHA256
- [ ] Hashes recorded in hashes.txt
- [ ] Hashes entered into case_metadata.json
- [ ] All hashes verified without mismatch
- [ ] Hash verification documented in chain_of_custody.txt
- [ ] No files modified after hashing
- [ ] Backups created with verified hashes

### Chain of Custody Checklist
- [ ] Case opened with examiner name
- [ ] All capture dates/times recorded
- [ ] Hash verification completed and documented
- [ ] Storage location documented
- [ ] Access restrictions noted
- [ ] Examiner signature required (if physical)
- [ ] All transitions documented

## Importing into Court Systems

When submitting evidence to court:

1. **Export Structure**
   - Maintain complete folder structure
   - Include all documentation files
   - Provide hashes.txt for verification
   - Include chain_of_custody.txt
   - Provide Final_Report.pdf

2. **Storage Media**
   - Burn to CD/DVD with verification
   - Use write-once media
   - Create hash of entire disk
   - Label with case number

3. **Documentation Package**
   - Original evidence folder structure
   - case_metadata.json for reference
   - Hash inventory (hashes.txt)
   - Chain of custody documentation
   - Forensic report (PDF)
   - Examiner credentials
   - Verification procedures used
